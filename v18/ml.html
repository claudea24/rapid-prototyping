ML Mock Interview

4 questions with follow-ups · ~10 minutes

Q1
How would you predict customer churn? Walk me through your approach.
A) Train a deep neural network on all available features — it'll find the patterns.
B) Start with logistic regression or gradient-boosted trees. Define churn clearly, engineer features like recency/frequency, validate with proper time-based splits.
C) Use a random forest with all the customer data and tune hyperparameters.
↪
Why not start with a neural network? Aren't they more powerful?
❌ Mediocre Answer

"Neural nets need too much data. Logistic regression is simpler."

✅ Great Answer

"Three reasons: (1) Interpretability — stakeholders need to understand WHY customers churn to take action, and tree-based models give feature importances. (2) Tabular data — gradient-boosted trees (XGBoost, LightGBM) consistently outperform neural nets on structured/tabular data. (3) Iteration speed — simpler models train fast, letting you iterate on feature engineering, which usually matters more than model complexity. Start simple, establish a baseline, then add complexity only if needed."

Q2
Precision vs recall — when would you optimize for each?
A) Precision when false positives are costly, recall when false negatives are costly.
B) Precision for spam filters (don't lose real emails). Recall for disease screening (don't miss sick patients). The business context determines the tradeoff.
C) Always optimize for accuracy — it balances both.
↪
When would you use F1 score instead of optimizing one or the other?
❌ Mediocre Answer

"F1 is always good because it balances precision and recall."

✅ Great Answer

"F1 is the harmonic mean of precision and recall — it penalizes extreme imbalance between them. Use it when: (1) you care equally about both, (2) classes are imbalanced (accuracy is misleading), or (3) as a single metric for model comparison. But often you DON'T want F1 — if missing cancer is 100x worse than a false alarm, use F2 (weighted F-beta) or just focus on recall with a minimum precision threshold. Always tie the metric to the business cost."

Q3
Your fraud detection model has 98% accuracy. Is that good?
A) Yes, 98% is excellent! Ship it.
B) Not necessarily — if only 2% of transactions are fraud, a model that predicts "not fraud" for everything gets 98% accuracy while catching zero fraud.
C) It depends on the baseline. We need to compare against a simple model.
↪
Exactly. So how do you handle class imbalance in this scenario?
❌ Mediocre Answer

"Use SMOTE to oversample the minority class, then train normally."

✅ Great Answer

"Multiple strategies: (1) Metrics — use precision-recall AUC, not accuracy. (2) Resampling — SMOTE for oversampling or random undersampling, but be careful with synthetic examples in test sets. (3) Class weights — set class_weight='balanced' to penalize misclassifying the rare class more. (4) Threshold tuning — don't just use 0.5; adjust the decision threshold to match business costs. (5) Anomaly detection — frame it as detecting outliers instead of classification. In practice, threshold tuning + class weights often outperform fancy resampling."

Q4
What is data leakage? Give me an example.
A) When your training data is too small and the model overfits.
B) When information from the test set (or the future) leaks into training, making the model look unrealistically good. Example: including "cancellation_date" as a feature to predict churn.
C) When you normalize the data before splitting into train/test.
↪
How does sklearn's Pipeline help prevent leakage?
❌ Mediocre Answer

"Pipeline just chains steps together for convenience."

✅ Great Answer

"Pipeline ensures preprocessing steps (scaling, imputation, encoding) are fit ONLY on training data and then applied to test data. Without it, you might fit a scaler on the entire dataset before splitting — the scaler has 'seen' test data statistics, which leaks information. Pipeline wraps everything so that during cross-validation, each fold independently fits the preprocessing. Two types of leakage: (1) Target leakage — features that encode the target (like cancellation date for churn). (2) Train-test contamination — preprocessing on full data before splitting. Pipeline prevents type 2."
