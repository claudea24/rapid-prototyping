Q1
Can you explain the logical execution order of a SQL query?
A) SELECT runs first, then FROM pulls the data, then WHERE filters it.
B) FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT
C) It depends on the database engine and query optimizer.
↪
Great. So why can't you use a SELECT alias in the WHERE clause?
❌ Mediocre Answer

"It's just a SQL syntax thing. You just can't do it."

✅ Great Answer

"Because WHERE is evaluated at step 2, before SELECT creates the alias at step 5. The alias literally doesn't exist yet when WHERE runs. You can reference it in ORDER BY, though, since that runs after SELECT."

Q2
What's the difference between RANK() and DENSE_RANK()?
A) They're basically the same thing, just different syntax.
B) RANK() skips numbers after ties (1,1,3), DENSE_RANK() doesn't (1,1,2).
C) RANK() is for ordering, DENSE_RANK() is for grouping.
↪
Good. When would you choose ROW_NUMBER() instead?
❌ Mediocre Answer

"ROW_NUMBER just gives unique numbers. Use it when you want unique IDs."

✅ Great Answer

"ROW_NUMBER() assigns a unique sequential integer regardless of ties — great for deduplication (pick one row per group) or pagination. RANK/DENSE_RANK preserve tie info, which matters when you need to know 'who tied for 2nd place.' A classic pattern: WHERE rn = 1 inside a CTE to get the latest record per user."

Q3
How would you find employees with no department?
A) Use a subquery: WHERE dept_id NOT IN (SELECT id FROM departments)
B) LEFT JOIN departments ON ... WHERE departments.id IS NULL
C) INNER JOIN and filter for NULLs.
↪
Nice. What if I need employees AND departments with no match on either side?
❌ Mediocre Answer

"Just do two LEFT JOINs and UNION them together."

✅ Great Answer

"Use a FULL OUTER JOIN. It returns all rows from both tables — matched rows plus unmatched from each side. Not all databases support it (MySQL doesn't natively), so a workaround is LEFT JOIN UNION ALL RIGHT JOIN WHERE left.id IS NULL. The key insight: LEFT JOIN + IS NULL finds unmatched from one side; FULL OUTER finds unmatched from both."

Q4
Write a query for month-over-month revenue change. How would you approach this?
A) Self-join the table on month = month - 1 using a subquery.
B) Use a CTE to aggregate monthly revenue, then LAG() to get the prior month.
C) Create a temp table for each month and join them.
↪
Why CTE over a correlated subquery here?
❌ Mediocre Answer

"CTEs are just cleaner syntax. It doesn't really matter which you use."

✅ Great Answer

"A CTE + LAG() is more readable and usually more performant. The CTE aggregates once, then LAG() pulls the previous row — one scan. A correlated subquery re-executes for every row, scanning the table N times. Here's the pattern:"

WITH monthly AS (
  SELECT DATE_TRUNC('month', order_date) AS mo,
         SUM(revenue) AS rev
  FROM orders GROUP BY 1
)
SELECT mo, rev,
  LAG(rev) OVER (ORDER BY mo) AS prev_rev,
  ROUND((rev - LAG(rev) OVER (ORDER BY mo))
    / LAG(rev) OVER (ORDER BY mo) * 100, 1) AS pct_change
FROM monthly ORDER BY mo;
Q5
This query is running slow. How do you debug it?
A) Just add more indexes on every column.
B) Run EXPLAIN/EXPLAIN ANALYZE, check for full table scans, then look at indexes and filter order.
C) Break it into smaller queries and run them separately.
↪
What would EXPLAIN show you? Walk me through what you'd look for.
❌ Mediocre Answer

"EXPLAIN shows you the query plan. You look for anything that seems slow."

✅ Great Answer

"EXPLAIN shows the query execution plan — the steps the optimizer chose. I look for: (1) Sequential/full table scans on large tables (means no useful index). (2) High estimated row counts at each step. (3) Nested loops on large datasets (might need a hash join). (4) Sort operations without indexes. Key fixes: add indexes on WHERE/JOIN columns, filter early (push predicates down), avoid SELECT *, and check if statistics are stale. EXPLAIN ANALYZE actually runs the query and shows real vs estimated rows — big gaps mean stale stats."
