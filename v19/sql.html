SQL — Company Interview Patterns

Each section shows which companies emphasize that topic and how they test it.

1.1 Query Execution Order GoogleAmazon

▾
Google frequently tests execution order understanding. Amazon asks "why doesn't my alias work in WHERE?"

When you write a SQL query, you begin with SELECT, but the database engine follows its own logical execution order. Understanding this order is the single most important step toward writing correct, efficient SQL.

StepClauseWhat It Does
1FROM / JOINIdentify source tables and perform joins
2WHEREFilter individual rows before grouping
3GROUP BYCollapse rows into groups
4HAVINGFilter groups after aggregation
5SELECTEvaluate expressions and aliases
6ORDER BYSort the result set
7LIMITRestrict the number of returned rows
SELECT   customer_id,
         SUM(amount) AS total_spent
FROM     orders
WHERE    order_date >= '2024-01-01'
GROUP BY customer_id
HAVING   SUM(amount) > 1000
ORDER BY total_spent DESC
LIMIT    10;
Key Insight

HAVING must use SUM(amount) rather than the alias total_spent, because the alias is not created until SELECT runs in step 5. Some databases relax this rule, but standard SQL does not.

1.2 Joining Tables MetaGoogleAmazon

▾
Every company tests JOIN knowledge. Meta asks about self-joins for social graph queries.

Real-world data rarely lives in a single table. Joins reunite related pieces of information. An INNER JOIN returns only matched rows. A LEFT JOIN returns every row from the left table, filling NULLs where no match exists.

Join TypeKeepsNon-MatchesUse When
INNER JOINOnly matched rowsDropped from bothRecords in both tables
LEFT JOINAll left + matched rightRight fills with NULLComplete roster from left
RIGHT JOINAll right + matched leftLeft fills with NULLMirror of LEFT
FULL OUTERAll rows from bothNULLs on both sidesReconciliation
CROSS JOINEvery row x every rowN/A (Cartesian)Generate combinations
-- INNER JOIN: only matched rows
SELECT e.name, d.dept_name
FROM   employees e
INNER JOIN departments d
  ON e.dept_id = d.dept_id;

-- LEFT JOIN: all employees, even unmatched
SELECT e.name, d.dept_name
FROM   employees e
LEFT JOIN departments d
  ON e.dept_id = d.dept_id;
Tip

A common interview pattern: find rows with no match using LEFT JOIN + WHERE d.dept_id IS NULL. More readable than NOT EXISTS.

1.3 Window Functions GoogleMeta

▾
Google loves window function problems. Meta uses them for ranking content/engagement metrics.

A window function performs a calculation across a "window" of rows defined by an OVER clause. The rows are not collapsed — each row carries an additional computed value reflecting its context.

FunctionTiesGaps After TiesExample (95k, 95k, 88k)Use Case
ROW_NUMBER()Broken arbitrarilyN/A1, 2, 3Deduplication, top-N per group
RANK()Same rankYes1, 1, 3Competitive standing
DENSE_RANK()Same rankNo1, 1, 2Consecutive labels
SELECT
  name, department, salary,
  ROW_NUMBER() OVER (PARTITION BY department ORDER BY salary DESC) AS row_num,
  RANK()       OVER (PARTITION BY department ORDER BY salary DESC) AS rnk,
  DENSE_RANK() OVER (PARTITION BY department ORDER BY salary DESC) AS d_rnk
FROM employees;
Tip

Beyond ranking, window functions support running totals (SUM() OVER), moving averages (AVG() OVER (ROWS BETWEEN ...)), and access to other rows via LAG() and LEAD().

1.4 Aggregation & Grouping AmazonMeta

▾
Amazon asks GROUP BY questions with tricky NULL handling. Meta tests WHERE vs HAVING.

WHERE filters individual rows before grouping — it cannot reference aggregate functions. HAVING filters after grouping, so it can.

-- This FAILS: WHERE cannot use aggregates
SELECT   department, COUNT(*) AS headcount
FROM     employees
WHERE    COUNT(*) > 5      -- ERROR!
GROUP BY department;

-- This WORKS: HAVING filters after aggregation
SELECT   department, COUNT(*) AS headcount
FROM     employees
GROUP BY department
HAVING   COUNT(*) > 5;
FunctionCounts NULLs?Returns
COUNT(*)Yes — counts all rowsTotal rows in group
COUNT(col)No — skips NULLsNumber of non-NULL values
SUM(col)Skips NULLsSum of non-NULL values
AVG(col)Skips NULLsAverage of non-NULL values
Key Insight

Every column in the SELECT clause must either appear in GROUP BY or be wrapped in an aggregate function. MySQL may silently pick an arbitrary value — almost always a bug.

1.5 Common Table Expressions Google

▾
Google tests complex multi-CTE queries. They want clean, readable SQL.

A CTE is a named, temporary result set introduced with WITH. CTEs let you break complex queries into digestible, named steps — like assigning intermediate results to variables.

WITH monthly_revenue AS (
  SELECT
    DATE_TRUNC('month', order_date) AS month,
    SUM(amount) AS revenue
  FROM   orders
  GROUP BY 1
),
with_prior AS (
  SELECT
    month, revenue,
    LAG(revenue) OVER (ORDER BY month) AS prev_revenue
  FROM monthly_revenue
)
SELECT
  month, revenue, prev_revenue,
  ROUND(100.0 * (revenue - prev_revenue) / prev_revenue, 1) AS pct_change
FROM   with_prior
ORDER BY month;
Tip

In most databases, CTEs are optimised the same way as subqueries. In PostgreSQL 12+, you can add MATERIALIZED or NOT MATERIALIZED to control this.

1.6 Query Optimisation AmazonGoogle

▾
Amazon cares deeply about query performance at scale. Expect EXPLAIN questions.

Writing correct SQL is necessary but not sufficient. A few principles go a long way: use EXPLAIN, index filter/join columns, avoid SELECT *, filter early, and prefer EXISTS over IN.

PrincipleWhyImpact
Use EXPLAINShows query planPrimary diagnostic tool
Index filter/join columnsO(n) → O(log n)Orders of magnitude faster
Avoid SELECT *Reads all columnsLess I/O, faster queries
Filter earlyPush WHERE before JOINReduces join size
EXISTS over INEXISTS short-circuitsFaster for large subqueries
EXPLAIN ANALYZE
SELECT * FROM orders
WHERE customer_id = 42;
Key Insight

Optimisation is about reducing the data the engine must read at each step. When a query is slow, ask: "Where is the engine reading more data than it needs to?"
