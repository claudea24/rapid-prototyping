Challenge 1 — Query Order Fix

Easy
Time:00:00

This query fails. The alias total_spent can't be used in WHERE because WHERE executes before SELECT. Fix it using HAVING.

Hint: Replace WHERE with HAVING for filtering on aggregate results. HAVING runs after GROUP BY.
SELECT customer_id, SUM(amount) AS total_spent FROM orders GROUP BY customer_id HAVING SUM(amount) > 1000;

Challenge 2 — The Missing Employees

Easy
Time:00:00

Write a LEFT JOIN to find employees who have no department assigned. The employees table has dept_id which may be NULL, and the departments table has id and name.

Hint: Use LEFT JOIN on departments, then filter WHERE departments.id IS NULL.
SELECT e.name FROM employees e LEFT JOIN departments d ON e.dept_id = d.id WHERE d.id IS NULL;

Challenge 3 — Top 3 Per Department

Medium
Time:00:00

Use a window function to find the top 3 highest salaries per department. Table: employees(id, name, department, salary). Return name, department, salary, and rank.

Hint: Use a CTE with ROW_NUMBER() OVER (PARTITION BY department ORDER BY salary DESC), then filter where rn <= 3.
WITH ranked AS ( SELECT name, department, salary, ROW_NUMBER() OVER ( PARTITION BY department ORDER BY salary DESC ) AS rn FROM employees ) SELECT name, department, salary, rn FROM ranked WHERE rn <= 3;

Challenge 4 — Count the NULLs

Medium
Time:00:00

What's the difference between COUNT(*) and COUNT(email)? Write both queries on the users(id, name, email) table to demonstrate. Some users have NULL emails.

Hint: COUNT(*) counts all rows regardless of NULLs. COUNT(column) skips NULLs. You can also use COUNT(*) - COUNT(email) to find how many NULLs exist.
-- COUNT(*) counts ALL rows including NULLs SELECT COUNT(*) AS total_rows FROM users; -- COUNT(email) counts only non-NULL values SELECT COUNT(email) AS non_null_emails FROM users; -- Bonus: count the NULLs specifically SELECT COUNT(*) AS total_rows, COUNT(email) AS has_email, COUNT(*) - COUNT(email) AS null_emails FROM users;

Challenge 5 — Month over Month

Hard
Time:00:00

Write a CTE to calculate month-over-month revenue change. Table: revenue(month DATE, amount DECIMAL). Show current month, previous month, and the percentage change.

Hint: Use LAG(amount) OVER (ORDER BY month) in a CTE to get the previous month's amount, then calculate the percentage change in the outer query.
WITH monthly AS ( SELECT month, amount AS current_revenue, LAG(amount) OVER (ORDER BY month) AS prev_revenue FROM revenue ) SELECT month, current_revenue, prev_revenue, ROUND( (current_revenue - prev_revenue) * 100.0 / prev_revenue, 2 ) AS pct_change FROM monthly WHERE prev_revenue IS NOT NULL ORDER BY month;

Challenge 6 — Optimize This

Hard
Time:00:00

This query is slow. It uses a correlated subquery with IN on a large table. Rewrite it using EXISTS for better performance, and suggest what index to add.

Hint: EXISTS stops scanning once it finds the first match (short-circuits), while IN must build the full result set. Add a composite index on orders(customer_id, order_date, total).
-- Rewritten with EXISTS (short-circuits on first match) SELECT * FROM customers c WHERE EXISTS ( SELECT 1 FROM orders o WHERE o.customer_id = c.id AND o.total > 500 AND o.order_date >= '2025-01-01' ); -- Recommended index: -- CREATE INDEX idx_orders_lookup -- ON orders(customer_id, order_date, total);
