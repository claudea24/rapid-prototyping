Statistics

3 conceptual challenges on hypothesis testing, A/B testing, and sample size calculations. Each challenge has its own timer.

Challenge 1 — Design the A/B Test

Medium
Time:00:00

Think about it... Your PM wants to test a new checkout flow. Current conversion rate is 3.2%. They want to detect a 10% relative improvement (to 3.52%). How would you design this A/B test? What's your null and alternative hypothesis? What are potential pitfalls?

Hypotheses:
H0: p_new = p_old = 0.032 (no difference)
H1: p_new != p_old (two-sided test)

Design:
- Randomization unit: user (not session, to avoid same-user contamination)
- Metric: conversion rate (purchases / unique visitors)
- Significance level: alpha = 0.05
- Power: 80% (beta = 0.20)
- MDE: 0.32 percentage points (3.2% to 3.52%)

Sample size (per group): ~120,000 users. Formula: n = (Z_alpha/2 + Z_beta)^2 * (p1(1-p1) + p2(1-p2)) / (p2-p1)^2. At 3.2% baseline with 0.32pp MDE, you need large samples because the effect is small.

Pitfalls:
1. Peeking — checking results before reaching sample size inflates false positive rate
2. Novelty effect — users may engage more simply because it's new
3. Simpson's paradox — results may differ across segments (mobile vs desktop)
4. Network effects — if users influence each other, independence assumption breaks

Challenge 2 — Interpret This P-value

Easy
Time:00:00

Think about it... Your A/B test returns p = 0.03. Your colleague says "there's a 3% chance the new feature doesn't work." Is this correct? What does p = 0.03 actually mean? What would you say to a non-technical stakeholder?

Your colleague is WRONG. This is the most common misinterpretation of p-values.

What p = 0.03 actually means: "If there were truly no difference between the groups (H0 is true), the probability of observing data as extreme as (or more extreme than) what we observed is 3%."

It does NOT mean:
- "3% chance the null is true" (that's a Bayesian posterior)
- "97% chance the feature works"
- "The effect size is large"

For stakeholders: "If the new feature had zero effect, it would be very unlikely (3 in 100 chance) to see results this different just by random chance. This gives us confidence the effect is real, but we should also look at the effect SIZE and confidence interval to judge if it's practically meaningful."

Key nuance: Statistical significance != practical significance. A p-value of 0.03 with a 0.01% conversion lift isn't actionable even though it's "significant."

Challenge 3 — Calculate Sample Size

Hard
Time:00:00

Think about it... You need to run a test where the baseline metric is a continuous variable (average revenue per user = $45, standard deviation = $30). You want to detect a $3 increase with 80% power and 5% significance. How many users per group do you need? Show your work.

Formula for two-sample t-test:
n = 2 * (Z_alpha/2 + Z_beta)^2 * sigma^2 / delta^2

Values:
- Z_alpha/2 = Z_0.025 = 1.96 (two-sided, alpha=0.05)
- Z_beta = Z_0.20 = 0.842 (power=80%)
- sigma = $30 (standard deviation)
- delta = $3 (minimum detectable effect)

Calculation:
n = 2 * (1.96 + 0.842)^2 * 30^2 / 3^2
n = 2 * (2.802)^2 * 900 / 9
n = 2 * 7.851 * 100
n = 1,570.2

Answer: ~1,571 users per group (3,142 total).

Practical note: This assumes equal variance in both groups and no attrition. In practice, you'd add 10-20% buffer for drop-offs. If your site gets 1,000 users/day, you need about 3-4 days to run this test. The large sample is needed because the effect ($3) is small relative to the noise ($30 SD) — the signal-to-noise ratio (effect/SD = 0.1) is low.
