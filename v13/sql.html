Query Execution Order

▼

When you write a SQL query, you begin with SELECT, follow it with FROM, perhaps a WHERE clause. But the database does not read your query top to bottom. It follows its own logical execution order, and understanding that order is the single most important step toward writing correct SQL.

A common frustration: you create an alias in SELECT — say, revenue - cost AS profit — then filter on it in WHERE. The query fails. Why? Because WHERE is evaluated before SELECT. The alias doesn't exist yet.

1FROMIdentify source tables and perform joins
2WHEREFilter individual rows before grouping
3GROUP BYCollapse rows into groups
4HAVINGFilter groups after aggregation
5SELECTEvaluate expressions and aliases
6ORDER BYSort the result set
7LIMITRestrict the number of returned rows
Key Insight

HAVING must use SUM(amount) rather than the alias total_spent, because the alias is not created until SELECT runs in step 5. Some databases relax this rule, but standard SQL does not. Always write with the standard in mind.

Joining Tables

▼

Real-world data rarely lives in a single table. Joins reunite related pieces of information from separate tables. Think of two class rosters: an INNER JOIN returns only students in both classes; a LEFT JOIN keeps everyone from the first roster, filling NULLs for those not in the second.

Join TypeKeepsNon-MatchesUse When
INNER JOINOnly matched rowsDropped from both sidesYou only want records that exist in both tables
LEFT JOINAll left + matched rightRight fills with NULLYou need a complete roster from the left table
RIGHT JOINAll right + matched leftLeft fills with NULLMirror of LEFT — swap table order instead
FULL OUTERAll rows from bothNULLs on both sidesReconciliation — find records missing from either side
CROSS JOINEvery row × every rowN/A (Cartesian product)Generate combinations (e.g., date spines)
Interview Tip

A common pattern: find rows with no match. Use LEFT JOIN then filter with WHERE right.id IS NULL. This is often more readable than NOT EXISTS.

Window Functions

▼

Aggregate functions like SUM collapse rows. But what if you need a computation across related rows while keeping every row intact? Window functions perform calculations over a "window" of rows defined by OVER(), without collapsing anything.

Three ranking functions appear constantly in interviews:

FunctionTiesGaps After TiesExample (95k, 95k, 88k)Use Case
ROW_NUMBER()Broken arbitrarilyN/A — always unique1, 2, 3Deduplication, top-N per group
RANK()Same rankYes — skips next1, 1, 3Competitive standing (Olympics)
DENSE_RANK()Same rankNo — consecutive1, 1, 2Consecutive labels regardless of ties

Choosing: Need exactly one winner per partition (dedup)? Use ROW_NUMBER. Need competitive standing with gaps (gold, gold, bronze)? RANK. Need consecutive labels? DENSE_RANK.

Key Insight

Beyond ranking, window functions support running totals (SUM() OVER), moving averages (AVG() OVER (ROWS BETWEEN ...)), and access to other rows via LAG() and LEAD().

Aggregation & Grouping

▼

GROUP BY collapses rows sharing the same values into a single output row. The question that surfaces repeatedly is: what's the difference between WHERE and HAVING? The answer follows from the execution order: WHERE filters rows before grouping (can't use aggregates), HAVING filters after grouping (can use aggregates).

FunctionCounts NULLs?Returns
COUNT(*)Yes — counts all rowsTotal number of rows in group
COUNT(col)No — skips NULLsNumber of non-NULL values
SUM(col)Skips NULLsSum of non-NULL values
AVG(col)Skips NULLsAverage of non-NULL values
MIN(col) / MAX(col)Skips NULLsMinimum / maximum value
Key Insight

Every column in SELECT must either appear in GROUP BY or be wrapped in an aggregate function. If you select dept and name but only group by dept, the database doesn't know which name to display. Most databases raise an error; MySQL may silently pick an arbitrary value — almost always a bug.

Common Table Expressions (CTEs)

▼

A CTE is a named, temporary result set that exists only for the duration of a single query. Introduced with WITH, CTEs let you break complex queries into digestible, named steps — much like assigning intermediate results to variables in a programming language.

Without CTEs, you'd nest derived tables inside each other, quickly becoming unreadable. With CTEs, each logical step gets a clear name and can be verified independently:

WITH monthly_revenue AS (
  SELECT DATE_TRUNC('month', order_date) AS month,
         SUM(amount) AS revenue
  FROM   orders
  GROUP BY 1
),
with_prior AS (
  SELECT month, revenue,
         LAG(revenue) OVER (ORDER BY month) AS prev
  FROM   monthly_revenue
)
SELECT month, revenue, prev,
  ROUND(100.0 * (revenue - prev) / prev, 1) AS pct_change
FROM   with_prior;

The first CTE computes the aggregate. The second uses LAG to fetch the prior month. The final SELECT calculates percentage change. Each step is self-contained.

Tip

In most databases, CTEs are optimised the same way as subqueries — the planner inlines them. Focus on mastering the non-recursive form: chaining two or three CTEs to build up a result step by step.

Query Optimisation

▼

Writing correct SQL is necessary but not sufficient. In production, queries run against millions or billions of rows. A few principles go a long way:

PrincipleWhyImpact
Use EXPLAINShows the query plan — which indexes, sequential scans, row estimatesPrimary diagnostic tool
Index filter/join columnsTurns full table scan O(n) into index lookup O(log n)Orders of magnitude faster
Avoid SELECT *Forces reading all columns; especially costly in columnar storageLess I/O, faster queries
Filter earlyPush WHERE before JOIN to reduce rows entering the joinDramatically reduces join size
EXISTS over INEXISTS short-circuits on first match; IN evaluates entire subqueryFaster for large subqueries
Key Insight

Optimisation is about reducing the amount of data the engine must read at each step. Indexing, projection, early filtering — they all serve that single goal. When a query is slow, ask: "Where is the engine reading more data than it needs to?"
