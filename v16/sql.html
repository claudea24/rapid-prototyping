The Spotify Puzzle — SQL

A narrative journey through SQL fundamentals. You just joined Spotify's data team — and your first day is anything but quiet.

You just joined Spotify's data team. It's 9:15 AM on your second Monday when your manager rushes over: "Weekly active users dropped 15% — the VP wants answers by end of day. Here's access to our database. Good luck."

Scene 1 First Look — Query Execution Order

▾
You open the query editor and start typing. But your first query fails — you used an alias in the WHERE clause and got a cryptic error. Before diving deeper, you need to understand how SQL actually processes your queries. It's not top to bottom.

When you write SQL, the clauses appear as SELECT → FROM → WHERE, but the engine processes them in a different logical order. Understanding this explains why you can't use a SELECT alias in WHERE — the alias doesn't exist yet when WHERE runs.

1
FROM / JOIN
Identify source tables and perform joins
2
WHERE
Filter individual rows before grouping
3
GROUP BY
Collapse rows into groups
4
HAVING
Filter groups after aggregation
5
SELECT
Evaluate expressions and aliases
6
ORDER BY
Sort the result set
7
LIMIT
Restrict the number of returned rows

Let's trace through a concrete query. You have a table called orders with columns customer_id, amount, and order_date. You want to find customers who spent more than $1,000 in 2024:

SELECT   customer_id,
         SUM(amount) AS total_spent
FROM     orders
WHERE    order_date >= '2024-01-01'
GROUP BY customer_id
HAVING   SUM(amount) > 1000
ORDER BY total_spent DESC
LIMIT    10;

The engine proceeds: Step 1 (FROM): load the orders table. Step 2 (WHERE): discard rows before 2024. Step 3 (GROUP BY): group by customer_id. Step 4 (HAVING): remove groups with SUM ≤ 1000. Step 5 (SELECT): compute SUM(amount) and assign the alias. Step 6 (ORDER BY): sort descending. Step 7 (LIMIT): return top 10.

Written OrderExecution OrderWhy It Matters
SELECT (first written)5th to executeAliases aren't available until here
FROM (second written)1st to executeEngine starts by finding the data
WHERE (third written)2nd to executeCan't use aggregates — groups don't exist yet
HAVING (later written)4th to executeCAN use aggregates — groups exist now
Key Insight

HAVING must use SUM(amount) rather than the alias total_spent, because the alias isn't created until SELECT runs in step 5. Some databases (MySQL) relax this, but standard SQL does not.

Now you know why your alias failed in WHERE. You fix the query and pull the first results. The numbers confirm a real drop — but you'll need more tables to find out why. Time to join some data.

Scene 2 Joining the Clues — Joins

▾
The users table alone isn't enough. You need to join it with sessions and subscriptions to see the full picture. But which join type? If you use INNER JOIN, you'll silently drop users who never had a session this week — exactly the users you're looking for.

An INNER JOIN returns only rows that have a match in both tables. A LEFT JOIN returns every row from the left table, filling NULLs where no match exists. A FULL OUTER JOIN keeps all rows from both sides.

Join TypeKeepsNon-MatchesUse When
INNER JOINOnly matched rowsDropped from both sidesYou only want records in both tables
LEFT JOINAll left + matched rightRight fills with NULLNeed complete roster from left table
RIGHT JOINAll right + matched leftLeft fills with NULLMirror of LEFT — swap table order instead
FULL OUTERAll rows from bothNULLs on both sidesReconciliation — find records missing from either
CROSS JOINEvery row × every rowN/A (Cartesian product)Generate combinations (e.g., date spines)

Consider two small tables for an example:

employees: Alice (dept 10), Bob (dept 20), Carol (NULL). departments: 10=Engineering, 20=Marketing, 30=Finance.

-- INNER JOIN: only matched rows
SELECT e.name, d.dept_name
FROM   employees e
INNER JOIN departments d
  ON e.dept_id = d.dept_id;
-- Result: Alice|Engineering, Bob|Marketing
-- Carol excluded (NULL dept_id), Finance excluded (no employee)

-- LEFT JOIN: all employees, even unmatched
SELECT e.name, d.dept_name
FROM   employees e
LEFT JOIN departments d
  ON e.dept_id = d.dept_id;
-- Result: Alice|Engineering, Bob|Marketing, Carol|NULL
Key Insight

A common interview pattern: find rows with no match using LEFT JOIN + WHERE d.dept_id IS NULL. This is more readable (and equally efficient) than a NOT EXISTS subquery.

You use a LEFT JOIN from users to sessions — and immediately see thousands of rows where session data is NULL. These users didn't open the app at all this week. Now you need to rank which segments had the biggest drops.

Scene 3 Ranking the Suspects — Window Functions

▾
Your manager leans over: "Which user segments had the biggest drops? Can you rank them?" You need window functions — aggregate-like computations that don't collapse your rows. Each row keeps its identity but gains a ranking value.

A window function performs a calculation across a "window" of rows defined by an OVER clause. Unlike aggregates, the rows are not collapsed — you still get one output row for each input row.

FunctionTiesGaps After TiesExample (95k, 95k, 88k)Use Case
ROW_NUMBER()Broken arbitrarilyN/A — always unique1, 2, 3Deduplication, top-N per group
RANK()Same rankYes — skips next1, 1, 3Competitive standing (Olympics)
DENSE_RANK()Same rankNo — consecutive1, 1, 2Consecutive labels regardless of ties
 
SELECT
  name, department, salary,
  ROW_NUMBER() OVER (PARTITION BY department ORDER BY salary DESC) AS row_num,
  RANK()       OVER (PARTITION BY department ORDER BY salary DESC) AS rnk,
  DENSE_RANK() OVER (PARTITION BY department ORDER BY salary DESC) AS d_rnk
FROM employees;

Alice and Bob tie at $95,000 in Engineering. ROW_NUMBER assigns them 1 and 2 (arbitrary). RANK gives both 1, then skips to 3 for Carol. DENSE_RANK gives both 1, then proceeds to 2 for Carol.

Key Insight

Beyond ranking, window functions support running totals (SUM() OVER), moving averages (AVG() OVER (ROWS BETWEEN ...)), and access to other rows via LAG() and LEAD(). The LAG function will be crucial when we build the final report.

Your ranking query reveals that the biggest drop is concentrated among iOS users in the US and UK. Premium subscribers are barely affected. Now you need to aggregate the numbers to present a clean summary.

Scene 4 The Aggregation — GROUP BY

▾
Now you need to aggregate: how many users per platform, per week? You write a quick GROUP BY query, but it fails — you tried to use WHERE with COUNT(*). Time to learn the difference between WHERE and HAVING.

WHERE filters individual rows before grouping (cannot reference aggregates). HAVING filters after grouping (can reference aggregates). This follows directly from the execution order in Scene 1.

-- This FAILS: WHERE cannot use aggregates
SELECT   department, COUNT(*) AS headcount
FROM     employees
WHERE    COUNT(*) > 5      -- ERROR!
GROUP BY department;

-- This WORKS: HAVING filters after aggregation
SELECT   department, COUNT(*) AS headcount
FROM     employees
GROUP BY department
HAVING   COUNT(*) > 5;
FunctionCounts NULLs?Returns
COUNT(*)Yes — counts all rowsTotal rows in group
COUNT(col)No — skips NULLsNumber of non-NULL values
SUM(col)Skips NULLsSum of non-NULL values
AVG(col)Skips NULLsAverage of non-NULL values
Key Insight

Every column in SELECT must either appear in GROUP BY or be wrapped in an aggregate function. MySQL may silently pick an arbitrary value — almost always a bug. Always write with the standard in mind.

Your aggregated results are stark: iOS users dropped 22% while Android only dipped 3%. And it started exactly on Tuesday. Now you need to build a clean month-over-month comparison for the VP. Time for CTEs.

Scene 5 Building the Report — CTEs

▾
Time to build the final report. The VP wants a month-over-month comparison showing exactly when the drop started and how it compares to previous trends. You could nest subqueries inside subqueries — but that would be unreadable. CTEs let you break it into named steps.

A Common Table Expression (CTE) is a named, temporary result set introduced with WITH. It lets you break complex queries into digestible, named steps — like assigning intermediate results to variables in a programming language.

WITH monthly_revenue AS (
  SELECT
    DATE_TRUNC('month', order_date) AS month,
    SUM(amount) AS revenue
  FROM   orders
  GROUP BY 1
),
with_prior AS (
  SELECT
    month, revenue,
    LAG(revenue) OVER (ORDER BY month) AS prev_revenue
  FROM monthly_revenue
)
SELECT month, revenue, prev_revenue,
  ROUND(100.0 * (revenue - prev_revenue) / prev_revenue, 1) AS pct_change
FROM with_prior
ORDER BY month;

The first CTE, monthly_revenue, computes the aggregate. The second, with_prior, uses LAG to fetch the preceding month's revenue. The final SELECT calculates the percentage change. Each step is self-contained and easy to verify independently.

Key Insight

CTEs are also invaluable for recursive queries (traversing hierarchies), though recursive CTEs are less commonly tested in DS interviews. Focus on chaining two or three CTEs to build up a result step by step.

The report is clean and tells a clear story: a sharp week-over-week decline starting Tuesday, concentrated on iOS. But the query takes 45 seconds on production data. Your senior engineer says: "That won't fly for the VP demo. Optimize it."

Scene 6 The Presentation — Optimization + Resolution

▾
The query is taking too long on production data. Your senior tells you to run EXPLAIN and look for full table scans. "Every slow query has the same problem," she says. "It's reading more data than it needs to."

Writing correct SQL is necessary but not sufficient. A few optimization principles go a long way:

PrincipleWhyImpact
Use EXPLAINShows query plan — indexes, scans, row estimatesPrimary diagnostic tool
Index filter/join columnsTurns full scan O(n) into index lookup O(log n)Orders of magnitude faster
Avoid SELECT *Forces reading all columns; costly in columnar storageLess I/O, faster queries
Filter earlyPush WHERE before JOIN to reduce rows entering the joinDramatically reduces join size
EXISTS over INEXISTS short-circuits; IN evaluates entire subqueryFaster for large subqueries
 
EXPLAIN ANALYZE
SELECT * FROM orders
WHERE customer_id = 42;
Key Insight

Optimization is about reducing the amount of data the engine must read at each step. Every principle above — indexing, projection, early filtering — serves that single goal. When a query is slow, ask: "Where is the engine reading more data than it needs to?"

You add an index on the date column, push the WHERE clause inside the CTE, and drop SELECT *. Query time drops from 45 seconds to 1.2. The report loads instantly for the VP demo.
Story Resolution

You found it — an iOS app update (v4.7) introduced a crash on startup for users running iOS 16.x. The crash only affected the music player initialization, so users on newer iOS versions were fine. The drop started exactly when the update rolled out on Tuesday. The VP is impressed. The engineering team ships a hotfix by Wednesday. WAU recovers 12 of the 15 lost points within a week. Your manager says: "Not bad for your second week."
