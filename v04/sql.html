SQL

Queries, joins, window functions, and optimization

Query Execution Order#

Key

SQL doesn't execute top-to-bottom. The logical order is:

  1. FROM & JOINs — pick tables
  2. WHERE — filter rows
  3. GROUP BY — aggregate groups
  4. HAVING — filter groups
  5. SELECT — choose columns
  6. ORDER BY — sort results
  7. LIMIT / OFFSET — paginate

This is why you can't reference a SELECT alias in WHERE.

JOIN Types#

Key
  • INNER JOIN — only matching rows in both tables
  • LEFT JOIN — all rows from left + matches from right
  • RIGHT JOIN — all rows from right + matches from left
  • FULL OUTER JOIN — all rows from both, NULLs where no match
  • CROSS JOIN — Cartesian product of both tables
  • SELF JOIN — table joined to itself (e.g., manager hierarchy)

LEFT JOIN is the most common in interviews. Always consider NULL handling.

Window Functions#

Key

Compute values across a set of rows related to the current row — without collapsing groups.

SELECT
  employee_id,
  department,
  salary,
  RANK() OVER (
    PARTITION BY department
    ORDER BY salary DESC
  ) AS dept_rank
FROM employees;

RANK() skips positions on ties; DENSE_RANK() does not; ROW_NUMBER() is always unique.

Aggregation Gotchas#

Watch
  1. COUNT(*) vs COUNT(col) — COUNT(*) counts all rows; COUNT(col) ignores NULLs
  2. SELECT non-aggregated columns — every column in SELECT must be in GROUP BY or an aggregate
  3. HAVING vs WHERE — HAVING filters after grouping; WHERE filters before
  4. NULL in aggregates — SUM, AVG ignore NULLs; COUNT(*) does not
  5. Division by zero — guard with NULLIF(denominator, 0)

CTEs & Subqueries#

Tip

CTEs (Common Table Expressions) improve readability over nested subqueries.

WITH monthly_sales AS (
  SELECT
    user_id,
    DATE_TRUNC('month', order_date) AS mth,
    SUM(amount) AS total
  FROM orders
  GROUP BY user_id, mth
)
SELECT
  user_id, mth, total,
  LAG(total) OVER (
    PARTITION BY user_id ORDER BY mth
  ) AS prev_month
FROM monthly_sales;

CTEs can be referenced multiple times; subqueries cannot. Some engines materialize CTEs.

Performance Tips#

Tip
  1. Index columns used in WHERE, JOIN, and ORDER BY clauses
  2. Avoid SELECT * — fetch only the columns you need
  3. Use EXPLAIN / EXPLAIN ANALYZE to inspect the query plan
  4. Prefer EXISTS over IN for correlated subqueries (often faster)
  5. Partition large tables by date or category to reduce scan size

Indexing speeds reads but slows writes — balance based on workload.
