The Language of Data

Execution order, joins, window functions, and CTEs.

🗂 Logical Execution Order FOCUS

▼

The database does not read your query top to bottom. It follows a logical execution order, and understanding it is the most important step toward writing correct SQL.

StepClauseWhat It Does
1FROM / JOINIdentify source tables, perform joins
2WHEREFilter rows before grouping
3GROUP BYCollapse rows into groups
4HAVINGFilter groups after aggregation
5SELECTEvaluate expressions and aliases
6ORDER BYSort result set
7LIMITRestrict returned rows
Key Insight: HAVING must use SUM(amount) rather than the alias total_spent, because the alias is not created until SELECT runs in step 5.
SELECT   customer_id,
         SUM(amount) AS total_spent
FROM     orders
WHERE    order_date >= '2024-01-01'
GROUP BY customer_id
HAVING   SUM(amount) > 1000
ORDER BY total_spent DESC
LIMIT    10;

🔗 Joining Tables FOCUS

▼

Joins reunite related data spread across tables. The type of join determines which rows survive.

JoinKeepsUse When
INNEROnly matched rowsNeed records in both tables
LEFTAll left + matched rightComplete roster from left table
RIGHTAll right + matched leftMirror of LEFT
FULL OUTERAll rows from bothReconciliation
CROSSEvery row x every rowDate spines, combinations
Tip: Find unmatched rows with LEFT JOIN ... WHERE right.id IS NULL. Often more readable than NOT EXISTS.
-- All employees, even those without departments
SELECT e.name, d.dept_name
FROM   employees e
LEFT JOIN departments d
  ON e.dept_id = d.dept_id;

📈 Window Functions FOCUS

▼

Window functions compute a value across related rows while keeping every individual row intact — no collapsing.

FunctionTiesGapsExample (95k, 95k, 88k)
ROW_NUMBER()Broken arbitrarilyN/A1, 2, 3
RANK()Same rankYes1, 1, 3
DENSE_RANK()Same rankNo1, 1, 2
SELECT name, department, salary,
  ROW_NUMBER() OVER (PARTITION BY department ORDER BY salary DESC) AS rn,
  RANK()       OVER (PARTITION BY department ORDER BY salary DESC) AS rnk,
  DENSE_RANK() OVER (PARTITION BY department ORDER BY salary DESC) AS drnk
FROM employees;
Key Insight: Use ROW_NUMBER for deduplication (exactly one per group). Use RANK for competitive standing with gaps. Use DENSE_RANK for consecutive labels.

🗒 CTEs & Optimization FOCUS

▼

CTEs let you break complex queries into named, digestible steps. Think of them as variables in a programming language.

WITH monthly_revenue AS (
  SELECT DATE_TRUNC('month', order_date) AS month,
         SUM(amount) AS revenue
  FROM   orders
  GROUP BY 1
),
with_prior AS (
  SELECT month, revenue,
    LAG(revenue) OVER (ORDER BY month) AS prev_revenue
  FROM monthly_revenue
)
SELECT month, revenue, prev_revenue,
  ROUND(100.0 * (revenue - prev_revenue) / prev_revenue, 1) AS pct_change
FROM with_prior
ORDER BY month;

Optimization principles:

PrincipleWhy
Use EXPLAINShows query plan, indexes, scans
Index filter/join columnsO(n) scan becomes O(log n) lookup
Avoid SELECT *Less I/O, especially in columnar storage
Filter earlyReduce rows before joins
EXISTS over INShort-circuits for large subqueries

Execution Order

FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT

Cannot use SELECT alias in WHERE (runs before SELECT).

Joins

  • INNER: only matched rows
  • LEFT: all left + matched right (NULL for no match)
  • FULL OUTER: all rows from both sides
  • Find unmatched: LEFT JOIN ... WHERE right.id IS NULL

Window Functions

  • ROW_NUMBER() — unique, breaks ties (dedup)
  • RANK() — ties same, gaps after (1,1,3)
  • DENSE_RANK() — ties same, no gaps (1,1,2)
  • Also: LAG(), LEAD(), SUM() OVER

CTEs + Optimization

  • CTEs: named temporary result sets with WITH
  • Always use EXPLAIN to read query plans
  • Index WHERE/JOIN columns, filter early, avoid SELECT *
