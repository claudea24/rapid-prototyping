Case Studies Cheat Sheet

AARRR, frameworks, metric debugging, market sizing, and experiment design.

1 AARRR Pirate Metrics

Acquisition → Activation → Retention → Revenue → Referral
StageQuestionExample Metric
AcquisitionHow do users find us?Signups, CAC
ActivationGreat first experience?Onboarding completion
RetentionDo they come back?D7/D30 retention
RevenueDo they pay?ARPU, LTV
ReferralDo they tell others?Viral coefficient

Using AARRR in Interviews

When asked "how would you measure success for product X?", walk through each AARRR stage. This shows structured thinking. Focus on the stage most relevant to the company's current challenge (early-stage = acquisition, mature = retention/revenue).

2 Case Framework

Clarify → Metrics → Hypothesize → Analyze → Recommend → Measure
  • Clarify: restate, ask about scope, users, time
  • Metrics: pick primary + counter-metrics
  • Hypothesize: 3-4 possible explanations
  • Analyze: segment, compare, quantify
  • Recommend: action + expected impact
  • Measure: how to validate the recommendation

Example: "Engagement dropped 10% this week"

Clarify: Which metric? Which platform? Which user segment? Metrics: DAU, session length, actions per session. Hypothesize: (1) Bug/outage, (2) Seasonal, (3) Product change, (4) External event. Analyze: Check by platform, country, new vs returning, day-by-day. Recommend: If it's a bug, hotfix + post-mortem. If seasonal, adjust targets.

3 Metric Debugging

  • Data problem? Logging broken, definition changed
  • External? Seasonality, competitor, news event
  • Internal? Product change, bug, infra issue
  • Segment: platform, geo, cohort, new/existing
  • Quantify: how much does each factor explain?

Debugging Checklist

Always start with data quality. In interviews, saying "first I'd check if the data is correct" shows maturity. Then external factors (you can't control these). Then internal factors (you can act on these). Always segment to isolate the root cause.

Pro Tip

Decompose the metric: Revenue = Users x Conversion x ARPU. If revenue dropped, which component changed? This narrows your investigation immediately.

4 Market Sizing

ApproachMethodBest For
Top-downTAM → filter %Known market data
Bottom-upUnit → scale upNovel products
  • State assumptions explicitly
  • Round numbers for mental math
  • Sanity check: does the answer make sense?

Example: "How many piano tuners in Chicago?"

Bottom-up: Chicago pop ~3M. ~1M households. ~10% have pianos = 100K pianos. Tuned 1x/year. Tuning takes ~2hrs (including travel). A tuner works ~250 days x 4 tunings/day = 1000/year. 100K / 1000 = ~100 piano tuners.

Framework

Population → Relevant segment → Usage frequency → Supply capacity → Answer. Always show your work and reasoning, not just the number.

5 Experiment Design

  • Randomization: user-level (not session)
  • Duration: at least 1 week (capture weekly cycles)
  • Guardrails: revenue, latency, crash rate
  • Watch for: novelty effects, network effects
  • No peeking: pre-commit to analysis time

When A/B Testing Doesn't Work

Network effects: social features where treatment affects control (use cluster randomization). Two-sided markets: changing supply affects demand. Long-term effects: short test misses retention impact (use holdout groups). Small sample: not enough power (use Bayesian methods or proxy metrics).

Power Analysis

Before running: determine minimum detectable effect (MDE), baseline rate, significance level (α=0.05), and desired power (0.80). Calculate required sample size. If it takes too long to collect, consider a more sensitive metric.
