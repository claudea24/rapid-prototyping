Everything you need for SQL interviews on one screen.

1 Execution Order

FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT

⚠️ Can't use SELECT aliases in WHERE (evaluated before SELECT)

Why This Matters

SQL doesn't execute in the order you write it. Understanding execution order prevents common mistakes like referencing aliases too early or filtering aggregates in WHERE instead of HAVING.

-- This FAILS: alias not available in WHERE
SELECT salary * 1.1 AS new_salary
FROM employees
WHERE new_salary > 100000;

-- This WORKS: use the expression directly
SELECT salary * 1.1 AS new_salary
FROM employees
WHERE salary * 1.1 > 100000;
Key Insight

ORDER BY is the one place you CAN use SELECT aliases, because it runs after SELECT.

When to Use Each

INNER JOIN when you only want rows that exist in both tables. LEFT JOIN when you need all records from the primary table even if there's no match. FULL OUTER for reconciliation queries where you need to find mismatches in both directions.

-- Find customers with no orders (LEFT JOIN + NULL check)
SELECT c.name
FROM customers c
LEFT JOIN orders o ON c.id = o.customer_id
WHERE o.id IS NULL;

3 Window Functions

FunctionBehaviorUse For
ROW_NUMBERUnique sequentialDedup, top-N
RANKTies + gaps (1,1,3)Competition ranking
DENSE_RANKTies, no gaps (1,1,2)Consecutive ranking

Window Function Anatomy

function_name() OVER (
  PARTITION BY col    -- group rows
  ORDER BY col        -- sort within group
  ROWS BETWEEN ...    -- frame (optional)
)

Common Patterns

-- Running total
SUM(amount) OVER (ORDER BY date ROWS UNBOUNDED PRECEDING)

-- Moving average (3-day)
AVG(value) OVER (ORDER BY date ROWS BETWEEN 2 PRECEDING AND CURRENT ROW)

-- Dedup: keep latest per user
WITH ranked AS (
  SELECT *, ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY created_at DESC) AS rn
  FROM events
)
SELECT * FROM ranked WHERE rn = 1;

4 Aggregation Rules

  • WHERE = before grouping (no aggregates allowed)
  • HAVING = after grouping (can use aggregates)
  • COUNT(*) counts NULLs, COUNT(col) skips NULLs
  • Every non-aggregated SELECT column must be in GROUP BY

WHERE vs HAVING

-- WHERE filters rows BEFORE grouping
SELECT department, AVG(salary)
FROM employees
WHERE hire_date > '2020-01-01'   -- filter individual rows
GROUP BY department
HAVING AVG(salary) > 80000;      -- filter groups

COUNT Nuances

-- These return DIFFERENT results if col has NULLs:
SELECT COUNT(*)    FROM t;  -- counts all rows
SELECT COUNT(col)  FROM t;  -- counts non-NULL only
SELECT COUNT(DISTINCT col) FROM t;  -- unique non-NULL

5 CTE Pattern

WITH step1 AS ( SELECT ... ), step2 AS ( SELECT ... FROM step1 ) SELECT * FROM step2;

Why CTEs Over Subqueries

CTEs make complex queries readable by breaking them into named steps. They're evaluated once and referenced by name, like temporary views.

-- Multi-step analysis with CTEs
WITH daily_revenue AS (
  SELECT DATE(order_time) AS day, SUM(amount) AS rev
  FROM orders GROUP BY 1
),
with_growth AS (
  SELECT day, rev,
         LAG(rev) OVER (ORDER BY day) AS prev_rev,
         ROUND(100.0 * (rev - LAG(rev) OVER (ORDER BY day))
               / LAG(rev) OVER (ORDER BY day), 1) AS growth_pct
  FROM daily_revenue
)
SELECT * FROM with_growth WHERE growth_pct > 10;

6 Optimization Checklist

EXPLAIN ANALYZE Index filter/join cols No SELECT * Filter early EXISTS > IN Avoid functions on indexed cols

Reading EXPLAIN Output

Look for Seq Scan on large tables (might need index), Nested Loop with large tables (consider Hash Join), and high actual rows vs estimated rows (stale statistics).

EXISTS vs IN

-- EXISTS stops at first match (faster for large subqueries)
SELECT * FROM orders o
WHERE EXISTS (SELECT 1 FROM vip_customers v WHERE v.id = o.customer_id);

-- IN materializes the full subquery list first
SELECT * FROM orders
WHERE customer_id IN (SELECT id FROM vip_customers);
