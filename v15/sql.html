F1

Query Execution Order

+50 XP ▼

When you write a SQL query, you begin with SELECT, but the database engine does not read your query top to bottom. It follows its own logical execution order — understanding this is the single most important step toward writing correct SQL.

StepClauseWhat It Does
1FROM / JOINIdentify source tables and perform joins
2WHEREFilter individual rows before grouping
3GROUP BYCollapse rows into groups
4HAVINGFilter groups after aggregation
5SELECTEvaluate expressions and aliases
6ORDER BYSort the result set
7LIMITRestrict the number of returned rows
SELECT   customer_id,
         SUM(amount) AS total_spent
FROM     orders
WHERE    order_date >= '2024-01-01'
GROUP BY customer_id
HAVING   SUM(amount) > 1000
ORDER BY total_spent DESC
LIMIT    10;

The engine proceeds: FROM loads orders. WHERE discards rows before 2024. GROUP BY groups by customer_id. HAVING removes groups with ≤$1,000. SELECT computes the alias. ORDER BY sorts. LIMIT returns top 10.

Interactive: Execution Order Shuffle

Key Insight
HAVING must use SUM(amount) rather than the alias total_spent, because the alias is not created until SELECT runs in step 5. Always write with the standard in mind.
F2

Joining Tables

+50 XP ▼

Real-world data rarely lives in a single table. Joins reunite related pieces of information.

Join TypeKeepsNon-MatchesUse When
INNER JOINOnly matched rowsDropped from bothRecords in both tables
LEFT JOINAll left + matched rightRight fills with NULLComplete roster from left
RIGHT JOINAll right + matched leftLeft fills with NULLMirror of LEFT
FULL OUTERAll rows from bothNULLs on both sidesReconciliation
CROSS JOINEvery row × every rowN/A (Cartesian)Generate combinations
-- INNER JOIN: only matched rows
SELECT e.name, d.dept_name
FROM   employees e
INNER JOIN departments d ON e.dept_id = d.dept_id;

-- LEFT JOIN: all employees, even unmatched
SELECT e.name, d.dept_name
FROM   employees e
LEFT JOIN departments d ON e.dept_id = d.dept_id;

With INNER JOIN, Carol (NULL dept_id) is excluded. With LEFT JOIN, Carol appears with NULL department name — the left table's row is preserved.

Pro Tip
Find rows with no match: use LEFT JOIN + WHERE d.dept_id IS NULL. Often more readable than NOT EXISTS.
F3

Window Functions

+75 XP ▼

Window functions compute a value across a "window" of rows without collapsing them — you still get one output row per input row.

FunctionTiesGapsExample (95k, 95k, 88k)Use Case
ROW_NUMBER()Broken arbitrarilyN/A1, 2, 3Deduplication, top-N
RANK()Same rankYes — skips1, 1, 3Competitive standing
DENSE_RANK()Same rankNo — consecutive1, 1, 2Consecutive labels
SELECT
  name, department, salary,
  ROW_NUMBER() OVER (PARTITION BY department ORDER BY salary DESC) AS row_num,
  RANK()       OVER (PARTITION BY department ORDER BY salary DESC) AS rnk,
  DENSE_RANK() OVER (PARTITION BY department ORDER BY salary DESC) AS d_rnk
FROM employees;
Pro Tip
Beyond ranking, window functions support running totals (SUM() OVER), moving averages (AVG() OVER (ROWS BETWEEN ...)), and access to other rows via LAG() and LEAD().
F4

Aggregation & Grouping

+50 XP ▼

WHERE filters individual rows before grouping — it cannot reference aggregate functions. HAVING filters after grouping, so it can.

-- This FAILS: WHERE cannot use aggregates
SELECT department, COUNT(*) AS headcount
FROM   employees
WHERE  COUNT(*) > 5      -- ERROR!
GROUP BY department;

-- This WORKS: HAVING filters after aggregation
SELECT department, COUNT(*) AS headcount
FROM   employees
GROUP BY department
HAVING COUNT(*) > 5;
FunctionCounts NULLs?Returns
COUNT(*)Yes — counts all rowsTotal rows in group
COUNT(col)No — skips NULLsNon-NULL values
SUM(col)Skips NULLsSum of non-NULL values
AVG(col)Skips NULLsAverage of non-NULL values
Key Insight
Every column in SELECT must either appear in GROUP BY or be wrapped in an aggregate function. MySQL may silently pick an arbitrary value — almost always a bug.
F5

Common Table Expressions

+75 XP ▼

A CTE is a named, temporary result set that exists only for a single query. Introduced with WITH, CTEs break complex queries into digestible, named steps.

WITH monthly_revenue AS (
  SELECT
    DATE_TRUNC('month', order_date) AS month,
    SUM(amount)                      AS revenue
  FROM   orders
  GROUP BY 1
),
with_prior AS (
  SELECT
    month, revenue,
    LAG(revenue) OVER (ORDER BY month) AS prev_revenue
  FROM monthly_revenue
)
SELECT
  month, revenue, prev_revenue,
  ROUND(100.0 * (revenue - prev_revenue) / prev_revenue, 1) AS pct_change
FROM with_prior
ORDER BY month;

Each CTE is self-contained and easy to verify independently. CTEs are also invaluable for recursive queries (traversing hierarchies), though these are less commonly tested in DS interviews.

Pro Tip
In most databases, CTEs are optimised the same way as subqueries. In PostgreSQL 12+, you can add MATERIALIZED or NOT MATERIALIZED to control this.
F6

Query Optimisation

+100 XP ▼

Writing correct SQL is necessary but not sufficient. A poorly written query against millions of rows can take minutes instead of milliseconds.

PrincipleWhyImpact
Use EXPLAINShows query plan — indexes, scans, row estimatesPrimary diagnostic tool
Index filter/join columnsTurns full scan O(n) into O(log n)Orders of magnitude faster
Avoid SELECT *Forces reading all columnsLess I/O, faster queries
Filter earlyPush WHERE before JOINReduces join size
EXISTS over INEXISTS short-circuitsFaster for large subqueries
EXPLAIN ANALYZE
SELECT * FROM orders
WHERE customer_id = 42;
Key Insight
Optimisation is about reducing the data the engine must read at each step. Every principle — indexing, projection, early filtering — serves that single goal. When a query is slow, ask: "Where is the engine reading more data than it needs to?"
